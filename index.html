🌐 ¿Qué es la Conmutación en Telecomunicaciones?

Conmutación es la conexión que realizan los diferentes nodos que existen en distintos lugares y distancias para lograr un camino apropiado para conectar dos usuarios de una red de telecomunicaciones.

La conmutación permite la descongestión entre los usuarios de la red, disminuyendo el tráfico y aumentando el ancho de banda disponible.

📚 Conceptos Fundamentales

🔌 ¿Qué es una Red de Conmutación?

Es una red de telecomunicaciones especializada en el transporte de diferentes tipos de tráfico (voz, datos, video) mediante el establecimiento de caminos o circuitos entre los usuarios.

📞 Red de Conmutación de Circuitos

Especializada en el transporte de tráfico de voz tradicional. Ejemplo clásico: la red telefónica pública conmutada (PSTN).

🌍 Internet como Red de Conmutación

Internet es el ejemplo más grande de red de conmutación de paquetes, donde la información se divide en pequeños bloques que viajan independientemente.

🏢 Evolución de la Voz Empresarial hacia Telefonía IP

Las empresas modernas han migrado desde la telefonía tradicional (PSTN) hacia soluciones de Telefonía IP de extremo a extremo. Esto implica:

  • Integración de voz, datos y video sobre una misma infraestructura IP.
  • Uso de teléfonos IP en oficinas locales y remotas.
  • Implementación de centrales telefónicas virtuales (como 3CX en la nube).
  • Reducción de costos al eliminar redes separadas para voz y datos.

💡 Dato clave: Básicamente, existen dos tipos de arquitecturas de redes de comunicación: conmutación de circuitos y conmutación de paquetes. Todo el estudio de redes modernas se basa en entender estas dos filosofías.

⚠️ La Crisis del Software y el Nacimiento de la Ingeniería de Software

La crisis del software surgió porque los proyectos de desarrollo se volvieron extremadamente complejos y los equipos no contaban con metodologías claras para manejarlos. La Ingeniería de Software nació como respuesta directa a esta crisis.

🔴 Manifestaciones de la Crisis del Software

⏰ Atrasos constantes en los plazos de entrega acordados con el cliente.
🐛 Mala calidad en el código fuente, resultando en gran cantidad de errores (bugs).
📄 Falta total o parcial de documentación del sistema desarrollado.
📅 Ausencia de planificación previa al inicio del desarrollo.
👥 Mal trabajo en equipo o funcionamiento deficiente del equipo de desarrollo.
💸 Gasto excesivo de tiempo y dinero, superando los presupuestos estimados.

🎯 Objetivo de la Ingeniería de Software

Desarrollar software de calidad dentro del presupuesto y tiempo estimados, atacando directamente las causas que provocaron la crisis del software.

💡 Para Recordar

La Ingeniería de Software no es solo programar, es todo el proceso sistemático y disciplinado que abarca desde la concepción de la idea hasta el mantenimiento del producto final, asegurando que se cumplan los atributos de calidad esperados.

🔄 Ciclo de Vida del Software

Son las 3 grandes etapas que describen todo el proceso de creación y mantenimiento de un software, desde su concepción hasta su retiro.

📋 Etapa de Definición (Análisis Previo)

Aquí se "gesta" el software. Se plantean las razones para crearlo (inspiración, necesidad, petición del cliente). Se define qué se quiere construir en forma de requerimientos.

También se conoce como la etapa de elicitación (descubrir requerimientos).

Incluye:

  • 📊 Modelo de Negocio
  • 🎯 Definición del Problema
  • 📋 Requerimientos (RF y RNF)
  • 📏 Estimación
  • 🔍 Análisis de Factibilidad
  • 📅 Planificación

⚙️ Etapa de Desarrollo

Etapa destinada a realizar el desarrollo o implementación del software mediante metodologías establecidas.

Pasos internos:

  • 📝 Análisis detallado
  • 💻 Desarrollo y codificación
  • 🚀 Implementación
  • 🧪 Pruebas (Testing)

Tecnologías utilizadas: Git/GitHub, Next.js, TailwindCSS, MongoDB (ejemplo del proyecto).

🔧 Etapa de Mantenimiento

La última etapa del ciclo de vida. Busca prolongar la vida útil del software recién lanzado.

3 Tipos de Mantenimiento:

  • Preventivo: Corregir errores potenciales antes de que ocurran.
  • Correctivo: Corrección de errores actuales detectados.
  • Perfectivo: Agregar nuevas funcionalidades solicitadas por el cliente.

⭐ Atributos de Calidad del Software

Aspectos indispensables para considerar que un software funcional es de calidad. No todos tienen que estar presentes obligatoriamente (por ejemplo, la seguridad no siempre aplica en todos los contextos).

01 Correctitud

El programa debe hacer correctamente lo que debería hacer, sin errores de lógica. Es el atributo más fundamental.

02 Usabilidad

Debe ser fácil de usar para los usuarios finales. La interfaz debe ser intuitiva y amigable.

03 Seguridad

No siempre obligatoria, pero necesaria en sistemas con datos sensibles como banca, salud o información personal.

04 Mantenibilidad

El código debe ser fácil de modificar, corregir y extender en el futuro sin introducir nuevos errores.

05 Confiabilidad

El sistema debe funcionar consistentemente sin fallar cuando se le necesita, en cualquier circunstancia.

06 UX / User Experience

Diseño de interfaces e interacción para maximizar la usabilidad del sistema y la satisfacción del usuario.

🔍 Análisis Previo (Artefactos)

📊 Modelo de Negocio

Diagnóstico de la situación actual. Es el primer punto de partida de cualquier proyecto de software.

  • Se analiza el entorno actual y se visualiza el potencial producto.
  • No se debe intervenir el sistema actual todavía, solo observar y describir la forma de trabajo.

🔑 Clave: Observar cómo trabajan los usuarios antes de proponer soluciones tecnológicas.

🎯 Definición del Problema

Saber identificar, describir y entender el problema o las dificultades a afrontar.

  • Se usan técnicas como el Customer Pain (molestias del cliente).
  • Cada problema identificado debe ser específico y atómico.

📌 Ejemplo en Moodle: "Falta de auto-corrección ortográfica" → problema puntual con posibles soluciones concretas.

📋 Requerimientos

Modelar y plasmar las posibles características o formas de solucionar el problema con el sistema.

Se describen las funcionalidades o características que el programa tendrá que proveer.

📄 Resultado: Documento de Especificación de Requerimientos (Backlog).

🔎 Técnicas de Recolección de Información

👀 Observación: Ir al lugar y ver cómo trabajan los usuarios en su entorno natural.
🎤 Entrevistas: Pueden ser dirigidas (estructuradas) o no dirigidas (abiertas).
📝 Cuestionarios: Más casual y permite llegar a más personas simultáneamente.
🔬 Sistemas similares: Analizar productos paralelos como referencia de funcionalidades.

⚠️ Recuerda: Hay que identificar tanto usuarios directos (los que usan el sistema) como usuarios indirectos (los que reciben los resultados del sistema).

⚠️ Técnica Especial: "Dar la Vuelta al Calcetín"

Consiste en preguntarse: ¿Qué puede ir mal?

  1. Hacer un listado de todo lo que podría resultar mal.
  2. Identificar soluciones o acciones de prevención para cada posible problema.
  3. Plantear términos y condiciones de seguridad basados en esos riesgos.

Con esa lista de problemas + soluciones, ya se está haciendo ingeniería de requerimientos.

📋 Requerimientos: Funcionales (RF) y No Funcionales (RNF)

🔑 Diferencia Conceptual Clave

Feature (Característica)

Es lo que TIENE el sistema. Una propiedad o capacidad inherente del software.

Requerimiento

Característica o restricción del sistema propuesto. Lo que el sistema DEBE hacer o CÓMO debe ser.

El sistema actual es el CONTEXTO; el sistema propuesto es la SOLUCIÓN.

✅ Requerimientos Funcionales (RF)

Identificadores o descriptores de funcionalidades y servicios del sistema. Se refieren a acciones disponibles para el usuario.

Se escriben como un verbo (acción del sistema), generalmente en forma imperativa.

📌 Ejemplos en Word:

  • RF1: Imprimir Documento
  • RF2: Insertar Imagen
  • RF3: Guardar Archivo

🚫 Requerimientos No Funcionales (RNF)

Identifican restricciones sobre el sistema propuesto. Se definen desde la perspectiva de calidad (usuario, contexto, etc.).

Se escriben como descripciones más completas, no como simples verbos.

📌 Ejemplo en Word:

RNF1: El editor deberá realizar una corrección automática de ortografía mientras el usuario escribe.

✅ Todo Requerimiento Debe Ser:

Claro

No ambiguo. Debe decir exactamente qué es lo que se quiere. Es lo que lo distingue de un atributo de calidad.

Completo

Debe tener descripciones sólidas y fuertes — no pueden faltar detalles importantes para su implementación.

Realizable

Los requerimientos deben poder hacerse eventualmente — no deben ser imposibles de implementar con la tecnología disponible.

🌐 Dimensiones de los RNF (Perspectivas de Calidad)

🌍 Contexto

¿Qué está haciendo el usuario con el programa? Cómo se pueden implementar funcionalidades en 2do plano para facilitar el uso. Ej: autocompletado en un editor.

👤 Cliente

Visión del cliente — qué busca con el desarrollo de ese sistema, respetando todo lo que desea obtener.

💎 Calidad

Consideración de los atributos de calidad: seguridad, correctitud, mantenibilidad, confiabilidad, etc.

🖥️ Usabilidad

Diseño del propio sistema: colores, tipografía, tamaño de botones, tamaño de letra, disposición de elementos, etc.

🔧 Hardware/Software

Especificaciones de equipos usados y tecnologías de desarrollo de software requeridas para el funcionamiento.
